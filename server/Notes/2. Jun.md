# Today's Learning Notes – File Upload Architecture (Node.js, Express, Multer)

Today I learned how file uploads work internally in Express applications.

Express can parse JSON and URL-encoded bodies, but it cannot parse `multipart/form-data` used for file uploads. Multer acts as middleware between the HTTP request and the controller. It reads the incoming file stream, validates file type and size, generates filenames, saves files to disk, and populates `req.file`.

I learned that Multer callbacks such as `destination(req, file, cb)`, `filename(req, file, cb)`, and `fileFilter(req, file, cb)` are configuration hooks called by Multer during upload processing. `req` is provided even when unused, allowing future access to request-specific data such as user IDs or roles.

I understood the purpose of Express's `next()` function. It is used to pass control to the next middleware or forward errors to centralized error handlers. In my current implementation, `next` is not used because responses are sent directly.

I refactored the architecture by moving Multer out of `LocalStorageProvider`. Multer is an HTTP concern, while storage providers should focus only on storing, deleting, and retrieving file metadata.

I also learned why `StorageProvider` exists as an abstract contract. It enforces consistent methods (`upload`, `delete`, `getMetadata`) across implementations such as Local Storage and Amazon S3, enabling easy replacement without changing business logic.

Finally, I learned that upload directories should be created during application startup rather than on every upload request.

1. PROBLEM TIMELINE (in order it happened)
----------------------------------------

STAGE 1: "No Inference Provider available for model timbrooks/instruct-pix2pix"
- Cause: The model timbrooks/instruct-pix2pix is old and no longer has ANY
  active inference provider on Hugging Face's router (fal, replicate, etc.
  all dropped it). Models on HF can lose serverless support over time even
  if the model page itself still exists.
- Fix: Switched model to black-forest-labs/FLUX.1-Kontext-dev, which is
  actively supported by multiple providers (fal-ai, Replicate, Runware,
  DataCrunch, TogetherAI, HF) and is the current recommended model for
  instruction-based image editing.
- How to check this myself next time:
  curl https://huggingface.co/api/models/<model-id>?expand=inferenceProviderMapping
  If inferenceProviderMapping is empty -> no provider will serve it.

STAGE 2: "Cannot read properties of undefined (reading 'arrayBuffer')"
- Cause: Code was sending the image as `data: imageBlob` in the
  imageToImage() call. The current @huggingface/inference SDK expects the
  image under the `inputs` key, not `data`. Because of this mismatch, the
  fal-ai provider's internal payload builder received `undefined` for the
  image and crashed trying to call .arrayBuffer() on it.
- Fix: Changed `data: imageBlob` -> `inputs: imageBlob`
- Lesson: When an SDK auto-selects a provider ("Defaulting to 'auto'..."),
  errors can surface from deep inside provider-specific code
  (e.g. fal-ai.js inside node_modules). Read the stack trace carefully -
  it often points to provider-specific glue code, not your own logic.

STAGE 3: Duplicate `model` key bug (caught before it caused a runtime issue)
- Cause: Two `model:` keys existed in the same object literal:
      model: "timbrooks/instruct-pix2pix",
      model: "black-forest-labs/FLUX.1-Kontext-dev",
  In JS object literals, the second key silently overwrites the first.
  No syntax error, but confusing and a future bug magnet.
- Fix: Removed the dead/duplicate line, kept only the Kontext-dev model key.

RESULT: After both fixes (inputs key + removing duplicate model key),
image-to-image generation started working end-to-end.

----------------------------------------
2. KEY TECHNICAL LEARNINGS
----------------------------------------

a) Hugging Face "Inference Providers" is a PROXY, not a single free service.
   - Your request goes: your code -> HF router -> 3rd party provider
     (fal-ai, Replicate, Together, etc.)
   - HF does NOT host every model anymore on its own infra ("hf-inference"
     provider is now mostly CPU-only / small models). Most image-gen models
     run on partner GPUs, which are billed.

b) "auto" provider selection can silently pick different providers between
   runs/models. Logs show this clearly:
      Defaulting to 'auto' which will select the first provider available...
      Auto selected provider: fal-ai
   This means provider-specific payload quirks can surface unpredictably -
   worth pinning a provider explicitly in production for consistency:
      inference.imageToImage({ ..., provider: "fal-ai" })

c) Different models accept different parameter shapes depending on provider
   (e.g. some providers expect image_url instead of a raw blob for
   instruction-edit models like Kontext). If a payload-shape error appears
   again, check the specific provider's expected schema (e.g. fal.ai docs
   for fal-ai/flux-pro/kontext) rather than assuming the generic SDK docs
   apply uniformly across providers.

d) Billing reality check (asked: "is it free for ~100 users/day?")
   - NOT unlimited free. HF gives monthly free credits, then pay-as-you-go,
     billed at the underlying provider's real rate (no HF markup).
   - Reference price point: FLUX.1 Kontext [pro] on fal = $0.04/image.
     Kontext-dev (open weight, what we're using) is generally cheaper but
     still metered per generation - exact rate depends on which provider
     auto-resolves the request.
   - Rough math for the business case discussed:
        100 users/day x 1 image  = ~3,000 images/month
        @ $0.01/img  -> ~$30/month
        @ $0.04/img  -> ~$120/month
     Retries/regenerations by users will multiply this further.
   - Action needed: check real usage/spend at
     https://hf.co/settings/inference-providers (per-model, per-provider
     breakdown is available there).

e) Stale comments in code can mislead future debugging.
   - The aiService.js docstring says "Uses Hugging Face free tier
     (30k tokens/month)" - this is left over from an earlier/different
     setup (likely a text LLM call) and does NOT reflect how image
     generation is actually billed (per-image/per-compute-time, not
     token-based). Needs to be corrected or removed.
